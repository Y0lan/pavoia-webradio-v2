## Problem Statement

GAENDE (Yolan) is a DJ, producer, and software engineer who runs a 9-stage 24/7 webradio from a Whatbox seedbox. The current setup — 9 MPD instances, a basic Node.js/React frontend, Plex for music library — works but looks like a prototype. The vision is to rebuild the entire web layer into a product where **the curation process IS the content**: every visitor sees the digging calendar, artist graph, stats, and the full depth of a hand-curated music collection. No algorithms, no curtain — radical transparency.

The primary audience is the PAVOIA festival community and DJ peers, with internet strangers as a secondary audience. The visual identity matters as much as the features — it needs to feel like a club at 4am, not a dashboard.

## What Makes This Cool

Nothing like this exists. Community radios (NTS, Rinse) have cultural credibility but zero curation transparency. Spotify Wrapped proved people love shareable music data, but it's algorithmic and annual. Self-hosted streamers (Navidrome, Airsonic) are personal Spotify replacements with no public-facing depth.

GAENDE combines:
- **Live multi-stage radio** (9 themed streams, crossfading between them)
- **Full curation transparency** (GitHub-style digging calendar, every stat public)
- **Deep music intelligence** (artist enrichment, force-directed graph, 12+ stat visualizations)
- **A strong personal brand** (violet-black palette, Clash Display, Anti-Algorithm identity, audio-reactive UI)

The combination is the innovation. A listener goes from "oh this track is good" → click the artist → see the similarity graph → discover 5 new artists → see that GAENDE added this track last Tuesday at 11pm after a 14-track digging session. That's a level of intimacy no algorithm offers.

## Constraints

- **Seedbox (Whatbox)**: Gentoo Linux, no root access, rootless Podman only
- **All ports > 1024**: No privileged ports available
- **Existing infrastructure stays**: 9 MPD instances (ports 14000-14008), Plex at 127.0.0.1:31711
- **Music stays on Whatbox**: Files can't be moved; bridge must run on the seedbox
- **Podman, not Docker**: Rootless, daemonless container runtime
- **One developer**: GAENDE builds this with AI assistance (Claude Code + gstack)

## Premises

1. **The existing Whatbox infrastructure stays.** MPD streams, Plex library remain. The Node.js server is replaceable. The rebuild is the web/API layer only.
2. **Podman is the container runtime.** All ports > 1024, systemd user units with `loginctl enable-linger` for persistence.
3. **Plex is the source of truth for music.** Tracks, playlists, metadata, cover art flow from Plex → bridge → enrichment → database.
4. **The v3.1 COMPLETE spec is directional.** ~260 features across 37 sections. GAENDE will iterate live once the product is running — the spec is a guide, not a contract.
5. **This is a complete rebuild.** Go bridge API, Postgres, Redis, Meilisearch, Next.js 15 frontend. The current React UI is retired.
6. **Everything lives on Whatbox.** Bridge, database, frontend — all co-located via Podman compose. The frontend could be split to Vercel later if needed, but start simple.

### Crossfade Definition

**Crossfade = client-side audio transition when a user switches between stages.** Two `<audio>` elements stream from two different MPD HTTP outputs. `GainNode` envelopes fade one out and the other in (5 curve options: equal-power, S-curve, exponential, linear, cut). This is a browser-side Web Audio API trick — the MPD instances are independent and unaware of each other. Intra-stream track transitions are handled by MPD natively (gapless playback).

### Real-time Transports

Two transports, each for a specific purpose:
- **WebSocket (`/ws`):** Bidirectional, high-frequency, per-stage. Carries `now_playing` events (every track change), `stage_status` (online/offline), and `new_track_added` (Plex sync). Client subscribes to specific stages. Used because the client needs to send subscription messages and receive targeted per-stage data.
- **SSE (`/api/events`):** Unidirectional, low-frequency, global. Carries `listeners` (aggregate counts, every 10s), `sync_update` (Plex sync completed), `enrichment` (artist enrichment status). Used because these are broadcast-to-all events that don't need client-to-server messaging. `EventSource` handles reconnect natively (browser auto-reconnects with `Last-Event-ID`).

If the seedbox's proxy doesn't support SSE cleanly, collapse both into the WebSocket connection — add `listeners` and `sync_update` message types to the WS protocol.

### Key Schema Decisions

- **`artist_relations`:** Adjacency list with weighted edges. Columns: `artist_id_a`, `artist_id_b`, `relation_type` (similar, same_label, same_genre), `weight` (0.0-1.0, from Last.fm similarity score), `source` (lastfm, musicbrainz). The artist graph reads this table directly — nodes are `artists`, edges are `artist_relations` filtered by `min_weight`.
- **`library_tracks`:** Stores genre, BPM, `camelot_key`, label, year directly (extracted from Plex metadata at sync time). Enrichment *may* fill gaps (e.g., missing BPM from MusicBrainz) but the primary source is Plex.
- **`wrapped_data`:** Materialized — computed by a scheduled job (monthly on the 1st, annually on Dec 31). Stored as JSONB so the frontend can render it without re-computing. `GET /api/wrapped/:year` reads from this table directly.

## Resilience & Error Handling

- **MPD connection loss:** Bridge maintains a persistent TCP pool with automatic reconnection (exponential backoff, 1s → 30s). Stage shows "reconnecting..." in UI. Other stages unaffected.
- **Plex unreachable during sync:** Sync worker retries 3x with 30s intervals, then logs failure and waits for next cycle (5 min). Stale data is served from Postgres — sync is not blocking.
- **Enrichment API rate limits / failures:** Each enrichment source has its own circuit breaker (trips after 5 failures in 60s, resets after 5 min). Artists with partial enrichment are served with what's available. Background worker retries failed enrichments on the next pass.
- **Redis down:** Bridge falls back to in-memory cache for now-playing data. API responses served directly from Postgres (slower but functional).
- **Meilisearch down:** Search falls back to Postgres `ILIKE` query (slower, no fuzzy). Search bar shows "limited search" indicator.
- **WebSocket disconnect:** Client auto-reconnects with exponential backoff. Missed now-playing events are caught up via REST poll on reconnect.
- **Stream buffering / network issues:** Audio engine detects `stalled` / `error` events on `<audio>` elements. Retries with fresh connection after 3s. Shows "buffering..." in player bar.

## Data Migration

The new system starts with a **fresh Postgres database** populated by the initial Plex sync. No historical play data is migrated from the current Node.js prototype — there is no structured play history to migrate (the current system doesn't persist it). Play history begins accumulating from the moment the new bridge goes live. The Plex library IS the historical record of curation — `added_at` timestamps from Plex populate the digging calendar retroactively.

## Monitoring & Observability

- **Health endpoint:** `GET /health` on the bridge returns status of all dependencies (MPD connections, Postgres, Redis, Meilisearch, Plex) with response times. Returns 200 if core services are up, 503 if critical dependency is down.
- **Structured logging:** Bridge logs in JSON to stdout (captured by `podman logs`). Log levels: error, warn, info. Key events: MPD reconnection, Plex sync results, enrichment failures, WebSocket connections.
- **Container health checks:** Each service in `podman-compose.yml` gets a `healthcheck` directive (bridge: `/health`, postgres: `pg_isready`, redis: `redis-cli ping`, meilisearch: `/health`).
- **Uptime check:** Simple external ping to the health endpoint (UptimeRobot free tier or similar). Alert via email if the radio goes down.

## Authentication Model

- **Admin token:** Single bearer token stored in `.env` (`ADMIN_TOKEN`). Passed in `Authorization: Bearer <token>` header.
- **Protected operations (admin-only):** force Plex sync, force artist re-enrichment, manual artist merge, stage control (start/stop/restart MPD), view enrichment queue status.
- **No rotation needed for v1** — single-user system. The token is only used by GAENDE via the admin panel.
- **All other routes are public.** No user accounts, no login, no sessions. This is intentional — radical transparency means everything is open.

### Critical Fixes from Outside Voice
1. **CORS on audio streams (SHOWSTOPPER):** MPD httpd doesn't send CORS headers. `createMediaElementSource()` requires them. **FIX: Bridge proxies audio streams via `GET /api/stream/:stageId`.** This also solves listener counting — bridge knows who's connected.
2. **Plex `addedAt` timestamps:** Sync worker MUST extract Plex's `addedAt` field, not use Postgres `DEFAULT now()`. Otherwise the digging calendar collapses to one day on initial sync.
3. **DATABASE_URL:** Bridge runs on host → connect to Postgres at `localhost:15432`, not `postgres:5432`.
4. **Artist aliasing:** Electronic music has dirty metadata (aliases, feat. credits, remixer ambiguity). No dedup strategy yet — flag for v1 implementation, accept some duplicate nodes in the graph.
5. **Re-listen file paths:** Bridge needs a mapping from Plex library paths to actual Whatbox filesystem paths (`~/files/Webradio/`). Define `MUSIC_BASE_PATH` in bridge config.
6. **`track_plays` growth:** After ~1M rows (~1 year), add a materialized view or summary table for trend queries. On-request is fine for v1.
7. **Font loading:** 4 custom fonts (Syne, JetBrains Mono, Space Mono, Instrument Serif) = 200-600KB. Use `font-display: swap`, preload critical weights, subset unused glyphs. Syne and Space Mono are on Google Fonts (CDN option). JetBrains Mono via JetBrains CDN or self-hosted.
